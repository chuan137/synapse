# Synapse Eval

The eval pipeline extracts trajectory data from completed Synapse tasks, evaluates agent behavior against per-role thresholds, and proposes rule patches when patterns of failure emerge.

---

## v2 Schema

Each case file is a JSON document in `tests/cases/task_<id>_<label>.json` matching this shape:

```json
{
  "schema_version": 2,
  "task_id": 115,
  "title": "UI polish: P0 hover text color + sticky green approved",
  "trigger_msg_id": 1554,
  "source_msg_id": null,
  "result_msg_id": 1556,
  "commit_sha": "084f672",
  "started_at": 1749478382000,
  "finished_at": 1749478514000,
  "total_duration_ms": 132000,
  "agents": {
    "developer:37": {
      "agent_id": "cec50b17:37",
      "role": "developer",
      "tools": {
        "Read":  { "calls": 8, "avg_ms": 95,  "p90_ms": 120, "errors": 0, "error_rate": 0.0 },
        "Edit":  { "calls": 3, "avg_ms": 110, "p90_ms": 130, "errors": 0, "error_rate": 0.0 },
        "Bash":  { "calls": 4, "avg_ms": 800, "p90_ms": 1200,"errors": 1, "error_rate": 0.25 }
      },
      "blocked_events": [],
      "messages_in": 2,
      "messages_out": 1,
      "active_duration_ms": 3940
    },
    "orchestrator:0": {
      "agent_id": "cec50b17:0",
      "role": "orchestrator",
      "tools": { ... },
      "blocked_events": [],
      "messages_in": 1,
      "messages_out": 2,
      "active_duration_ms": 1200
    }
  },
  "blocked_events": [],
  "label": "good",
  "raw": {
    "messages": [ ... ],
    "tool_metrics": [ ... ]
  }
}
```

`agents` keys use `<role>:<slot>` format. `raw` preserves the v1 `messages` and `tool_metrics` arrays for the critic/gate pipeline until they are migrated.

## Per-Role Threshold Model

Thresholds live in `src/eval/thresholds.json` (generated by `synapse eval --calibrate`):

```json
{
  "calibrated_at": "2026-06-09T...",
  "sample_size": { "developer": 23, "orchestrator": 21 },
  "by_role": {
    "developer":    { "tool_calls_p90": 25, "duration_ms_p90": 3894, "error_rate_max": 0.05, "has_commit": true },
    "orchestrator": { "tool_calls_p90": 15, "duration_ms_p90": 2531, "error_rate_max": 0.05 }
  },
  "task_level": { "traceability_score_max": 1 }
}
```

`duration_ms_p90` is `active_duration_ms` — the sum of tool-call durations for that agent within the task, not wall-clock time. Wall-clock time is `total_duration_ms` at the task level.

If `thresholds.json` is missing, the evaluator falls back to hardcoded defaults (see `evaluator.ts:FALLBACK_THRESHOLDS`).

Roles with fewer than 3 good-case samples are excluded from calibration to avoid noisy thresholds.

## Calibration Workflow

1. Ensure `tests/cases/` contains regenerated v2 case files (see Migration below).
2. Run calibration:
   ```sh
   synapse eval --calibrate [--percentile 90]
   ```
   This reads all `*_good.json` cases, groups agent-level stats by role, computes the p90 for each metric, and writes `src/eval/thresholds.json`.

3. Inspect the output — flag any surprises (very low `duration_ms_p90` means tools are fast; very high `tool_calls_p90` suggests a verbose role). The numbers are empirical truth; adjust the percentile if needed.

4. Run eval to confirm the new thresholds behave as expected:
   ```sh
   synapse eval --limit 200
   ```

## Migration: v1 → v2 Case Files

Existing case files generated before the v2 extractor lack the `agents` map and v2 fields. Regenerate them in one shot:

```sh
synapse eval --regenerate-all
```

This re-extracts every task whose case file exists on disk, upgrading each to v2 format. The task must still be in the DB (completed). Tasks that were deleted from the DB are left as-is (their case files remain unchanged).

After regeneration, re-run `--calibrate` since the new agent-level data will split `_default` into role buckets.

## Soft Signals

Soft signals are informational — they appear in `soft_failures[]` on `EvalResult` but do not fail the case. They feed the critic for pattern detection.

| Signal | Trigger |
|---|---|
| `error_rate_per_tool` | Tool error_rate > 10% for (agent, tool) |
| `redundant_reads` | Read tool called > 3 times by same agent |
| `confused_count` | blocked_events with category CONFUSED |
| `over_delegation` | Orchestrator agent with no `delegate_task` / `spawn_agent` calls |
| `under_delegation` | Orchestrator agent used Edit or Write |
| `idle_drift` | `total_duration_ms / sum(active_duration_ms) > 10` — wall-clock far exceeds tool activity |

`idle_drift` catches stuck workers: long wall-clock with little actual tool activity. Currently soft-only. Promote to a hard threshold once calibration data accumulates — use `wall_clock_ms_p90` in `thresholds.json` as the starting point.

## Tool Metrics Attribution

Tool call attribution uses two paths in priority order:

1. **FK path** (post-`12f3c41`): `SELECT * FROM tool_metrics WHERE task_id = ?`. Workers get `task_id` populated automatically via a cookie set on `delegate_task` and cleared on `report_done`. Orchestrators always have `task_id = NULL` by design (they interleave tasks).

2. **Time-window fallback** (legacy): `WHERE synapse_agent_id = ? AND timestamp BETWEEN started_at AND finished_at`. Used when FK returns 0 rows — covers pre-migration rows and orchestrator-only tasks.

Both paths are valid. Empty `agents` maps are valid v2 (orchestrator-only tasks produce no worker tool_metrics).

## Window Report

`synapse eval-window` aggregates per-task eval data across a time window and produces a structured markdown report.

```bash
synapse eval-window --since 7d        # last 7 days
synapse eval-window --since 24h       # last 24 hours
synapse eval-window --from 2026-06-01 --to 2026-06-08
synapse eval-window --since 7d --role developer
synapse eval-window --since 7d --output my-report.md
```

The report is written to `.synapse/reports/<timestamp>-window-<duration>.md` and contains: Summary, Per-role aggregates, Tool usage, Threshold breaches, Blocked events, Idle drift, and Recurring patterns sections.

Use the `/eval-window-report` skill to run the CLI and add a narrative interpretation layer.

Implementation: `src/eval/window.ts` (~270 lines). Uses FK attribution (`tool_metrics.task_id`) where available; falls back to time-window for legacy rows.
